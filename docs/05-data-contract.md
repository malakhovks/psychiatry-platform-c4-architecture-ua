# Дані та формальний контракт алгоритму

## 1. Основні доменні сутності

### AlgorithmDefinition

Незмінне опубліковане визначення алгоритму. Містить метадані, джерела, застосовність, згоду, кроки, правила переходів, формули, червоні прапорці, результати, маршрути, повторний супровід, політику ШІ, політику даних і тестові випадки.

### ScenarioSession

Стан виконання конкретної версії алгоритму. Містить контекст суб'єкта, згоди, відповіді, результати вимірювань, обчислення, прапорці, проєкт і підтверджене рішення, траєкторію, завдання та посилання на аудиторські події.

### ClinicalFact

Типізований факт із кодом, значенням, одиницею, часовим інтервалом, джерелом, якістю та статусом підтвердження. Первинна відповідь, автоматично вилучений проєкт і клінічно підтверджений факт є різними записами.

### MeasurementResult

Посилання на первинний сигнал або медіа, параметри отримання, результат контролю якості, версію конвеєра, похідні ознаки, невизначеність і клінічний статус.

### RiskFlag

Структурована критична умова з ідентифікатором правила, пріоритетом, доказовими полями, часом виявлення, потрібною дією, строком, станом доставки й результатом реагування.

### ClinicalDecision

Рішення уповноваженого фахівця, відокремлене від автоматичного проєкту. Містить автора, роль, час, підтверджені/відхилені елементи, причину змін і версію траєкторії.

## 2. Класи сховищ

| Сховище | Вміст | Особливості |
|---|---|---|
| Операційна база | Сесії, черги, завдання, блокування | Висока доступність, коротший строк для технічних даних |
| Клінічне сховище | Факти, результати, траєкторії, рішення | Посилений контроль, історичність і походження |
| Реєстр алгоритмів | Підписані версії визначень і тести | Незмінність, контроль цілісності, відкликання статусом |
| Медіасховище | ЕКГ, аудіо, відео, зображення | Окремі ключі, життєвий цикл, потоковий доступ |
| База знань | Протоколи й дозволені фрагменти RAG | Версія, джерело, дата чинності |
| Аудиторське сховище | Append-only події та хеші | Окремі права, контроль цілісності |
| Аналітичне сховище | Псевдонімізовані/агреговані набори | Відокремлення ідентифікаторів, контроль вторинної мети |
| Кеш | Короткоживучі технічні маркери | Без довготривалого клінічного змісту |

## 3. Версійність і походження

Кожний обчислений результат повинен посилатися на:

- `algorithmId` і точну `algorithmVersion`;
- ідентифікатори введених відповідей і вимірювань;
- `ruleId` або `formulaId`;
- версію словника/термінології;
- версію моделі й конвеєра, якщо застосовувався ШІ;
- версію шаблону й політики запобіжників;
- джерело знань для RAG;
- час і суб'єкта виконання;
- статус клінічного підтвердження.

## 4. Формати обміну

Зовнішній обмін реалізується через профільовані контракти. Рекомендовані типи FHIR-ресурсів залежать від локального профілю, але можуть охоплювати Questionnaire/QuestionnaireResponse, Observation, RiskAssessment, CarePlan, Task, ServiceRequest, Appointment, Consent, Provenance та AuditEvent. Зображення й частина біосигналів можуть використовувати DICOM або спеціалізовані контракти з посиланням із клінічного ресурсу.

Фактичний перелік профілів, кодових систем і мапінгів затверджується для кожної інтеграції. Внутрішня модель не повинна залежати від специфіки одного постачальника МІС.

## 5. Схема та приклади

Машинозчитувана JSON Schema доступна у `schemas/algorithm-definition.schema.json`. Приклад універсального визначення — `examples/algorithm-template.yaml`; каталог профілів — `examples/algorithm-catalog.yaml`; шаблон контрольних тестів — `examples/test-case-template.yaml`.
